# 测试驱动开发 (TDD)

测试驱动开发（Test-Driven Development, TDD）是一种软件开发方法论，其核心流程为“红 - 绿 - 重构”：先编写一个失败的测试（红），再编写最少量的代码使测试通过（绿），最后优化代码结构（重构）。

## 在本项目中的实践

在 [[2026-04-03-验收表单实现]] 计划中，TDD 被作为核心执行策略，具体体现在以下环节：

### 1. 网格映射逻辑
- **失败测试**: 定义点位 ID 与二维坐标的转换预期（如 `point 1` -> `row 1, col 1`）。
- **最小实现**: 编写基础的数学转换公式。
- **验证**: 确保边界值（如 `point 1200`）计算正确。

### 2. 数据库初始化
- **失败测试**: 断言数据库创建后应包含特定数量的种子数据（1200 点位，18000 结果记录）。
- **最小实现**: 编写 SQL 建表语句及循环插入逻辑。
- **验证**: 确认数据完整性。

### 3. API 行为校验
- **失败测试**: 模拟非法请求（如超出范围的点位 ID），期望返回 400 状态码。
- **最小实现**: 添加路由参数校验逻辑。
- **验证**: 确保系统健壮性。

## 优势

- **代码可靠性**: 确保每个功能点都有对应的测试覆盖。
- **设计驱动**: 迫使开发者在写实现前先思考接口设计和预期行为。
- **文档化**: 测试用例本身即为可执行的文档，说明系统应如何工作。
