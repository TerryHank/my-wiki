# 矩阵验收工作台设计规格

本文档定义了验收表单系统的重构设计规格，核心目标是将原有的“录表”模式转变为面向“异常登记”的矩阵工作台模式。

## 核心设计原则

1.  **默认成功，稀疏存储**：系统假设所有点位默认为“成功”，仅在数据库中记录“失败”状态及备注，以此优化存储和交互效率。
2.  **矩阵即导航**：右侧密集矩阵不仅是总览，更是主要的导航区域；左侧固定为当前组的 2x2 四宫格工作台。
3.  **蛇形遍历分组**：点位编号遵循蛇形规则（偶数行从左到右，奇数行从右到左），每 4 个点自动分为一组，确保左右视图逻辑一致。
4.  **场景隔离**：支持多场景动态切换，每个场景独立维护点位状态和备注，导出时仅针对当前场景。

## 关键交互规则

-   **状态切换**：单击点位在“成功/失败”间切换，失败时强制要求填写备注。
-   **备注保留**：点位恢复为“成功”后，历史备注不予删除，作为故障历史记录保留。
-   **动态矩阵**：矩阵行列数根据用户输入的总点数自动计算（`ceil(sqrt(n))`），而非固定尺寸。
-   **视觉反馈**：失败点显示为红色，当前组及当前点在矩阵中有明确的高亮层级区分。

## 技术架构约束

-   **前端**：继续使用 [[vitepress]] 作为应用壳子，承载 Vue 3 组件。
-   **后端**：Node.js 提供轻量 API，使用 [[sqlite]] 进行持久化。
-   **部署**：内网单机部署（`10.0.0.99`），无登录体系，无复杂权限控制。
-   **数据模型**：废弃旧版的“多次测试机会”模型，采用 `scene_point_records` 表记录异常。

## 待办与风险

-   需确保前后端共享同一套蛇形坐标计算逻辑，防止定位错位。
-   需处理自动保存失败时的状态回滚，避免界面与数据库不一致。
-   未来若需区分“未测试”与“确认成功”，需引入第三种状态，当前版本暂不支持。

# 蛇形遍历与分组规则

在矩阵验收系统中，为了符合人工巡检的自然路径并优化分组逻辑，采用蛇形遍历（Snake Traversal）算法生成点位编号和分组。

## 算法规则

1.  **坐标定义**：使用零基坐标 `(x, y)`，左上角为 `(0, 0)`，`x` 轴向右递增，`y` 轴向下递增。
2.  **遍历方向**：
    -   **偶数行**（`y = 0, 2, 4...`）：从左到右（`x: 0 → max`）。
    -   **奇数行**（`y = 1, 3, 5...`）：从右到左（`x: max → 0`）。
3.  **编号生成**：点号从 1 开始，严格按照上述遍历顺序递增。点号 1 永远位于 `(0, 0)`。
4.  **分组逻辑**：
    -   按遍历顺序，每 **4** 个连续点分为一组。
    -   第 1-4 点为第 1 组，第 5-8 点为第 2 组，依此类推。
    -   组内点在左侧工作台始终呈现为严格的 **2x2 四宫格** 布局。

## 应用场景

此规则主要用于解决大规模点位矩阵中的导航一致性问题。通过蛇形编号，相邻编号的点在物理空间上也是相邻的（除行尾换行处外），符合巡检人员的移动习惯。同时，基于遍历顺序的分组确保了右侧矩阵点击任意点后，左侧工作台区能准确展示其所在的局部上下文。

## 边界情况处理

-   **末尾空位**：当总点数无法填满最后一行时，剩余坐标位视为空位，不参与分组、不可点击、不导出。
-   **动态尺寸**：列数 `columnCount = ceil(sqrt(pointCount))`，行数 `rowCount = ceil(pointCount / columnCount)`，确保矩阵尽量接近正方形。

## 相关概念

-   [[grid-mapping]]：一维 ID 与二维坐标的相互转换基础。
-   [[acceptance-form-system]]：本规则的具体实施系统。

# 稀疏存储模式

一种针对“默认成功”场景的数据库设计优化策略。在验收系统中，绝大多数点位在正常情况下的状态均为“成功”，若为每个点位都创建记录会造成大量冗余数据。

## 核心逻辑

-   **隐式成功**：数据库中不存在记录时，系统默认该点位状态为“成功”。
-   **显式异常**：仅当点位状态为“失败”或存在“备注”时，才在 `scene_point_records` 表中创建或更新记录。
-   **记录保留**：即使点位状态从“失败”恢复为“成功”，若存在历史备注，记录仍保留在数据库中，不执行删除操作，以维持故障历史轨迹。
-   **清理规则**：仅当 `is_failed = 0` 且 `note = ''`（空字符串）时，才允许物理删除该条记录，回归隐式成功状态。

## 优势

1.  **存储效率**：在合格率高的场景下，数据库表大小极小，查询速度快。
2.  **交互简化**：前端无需加载全量状态，只需获取异常点列表即可渲染矩阵（默认渲染为成功色）。
3.  **历史追溯**：通过保留非失败状态的备注记录，实现了“状态恢复但历史留痕”的需求。

## 数据模型示例

```sql
CREATE TABLE scene_point_records (
    point_id INTEGER NOT NULL,
    scene_id INTEGER NOT NULL,
    is_failed INTEGER NOT NULL DEFAULT 0,
    note TEXT NOT NULL DEFAULT '',
    updated_at TEXT NOT NULL,
    PRIMARY KEY (point_id, scene_id)
);
```

## 潜在风险

-   **语义模糊**：需明确区分“无记录（默认成功）”与“有记录但状态为成功（历史遗留备注）”。前端展示和导出逻辑需对此做特殊处理，避免误读。
-   **统计复杂性**：计算总成功率时，需用 `总点数 - 失败记录数`，而非直接统计成功记录。

# 验收表单系统

验收表单系统（Acceptance Form System）是一套基于 Web 技术栈构建的单机内网工程验收录入工具。该系统旨在替代传统的 Excel 录入方式，提供可视化的矩阵导航和高效的异常登记流程。

## 系统架构

-   **前端框架**：利用 [[vitepress]] 作为应用容器，结合 Vue 3 构建响应式界面。
-   **后端服务**：Node.js 提供轻量级 RESTful API。
-   **数据存储**：使用 [[sqlite]] 嵌入式数据库进行本地持久化。
-   **部署环境**：部署于内网服务器（`10.0.0.99`），无需互联网访问，无用户登录体系。

## 核心功能特性

1.  **矩阵工作台**：
    -   右侧为密集点位矩阵总览，支持悬停显示详情。
    -   左侧为当前组 2x2 四宫格详细操作区。
    -   点位编号遵循 [[snake-traversal | 蛇形遍历规则]]。
2.  **异常登记模式**：
    -   采用 [[sparse-storage | 稀疏存储模式]]，默认所有点位成功，仅记录失败和备注。
    -   支持多场景（如不同天气、不同批次）独立切换与数据隔离。
3.  **自动化辅助**：
    -   根据输入总点数自动生成矩阵行列布局。
    -   操作实时自动保存，支持 Excel 导出当前场景数据。

## 发展历程

-   **V1 (旧版)**：采用“多次数测试方块”模型，交互类似传统表单录入，效率较低。
-   **V2 (重构版)**：2026-04-03 设计定稿，转向“异常登记”模型，强化矩阵导航和视觉反馈，废弃旧版多次测试逻辑。

## 技术亮点

-   在不依赖重型后端框架的前提下，实现了复杂的状态管理和数据持久化。
-   通过共享常量模块确保前后端在坐标计算和分组逻辑上的一致性。
-   实施严格的 [[test-driven-development | 测试驱动开发]] 流程，保证重构质量。
