# 验收表单交互系统实现计划

本文档是一份详细的技术实施计划，旨在构建一个基于 **VitePress**、**Node.js** 和 **SQLite** 的单机内网验收录入系统。该系统支持 1200 个点位、3 个场景、每场景 5 次点击录入，并提供右侧二维点阵总览和自动保存功能。

## 核心目标

构建一个轻量级、无需重型框架的全栈应用，利用 VitePress 作为应用壳子（App Shell），通过 Vue 3 组件承载业务逻辑，Node.js 原生 HTTP 服务提供 API，SQLite 进行本地持久化存储。

## 技术栈

- **前端框架**: Vue 3, VitePress (作为静态站点生成器及应用容器)
- **后端运行时**: Node.js (原生 HTTP 模块)
- **数据库**: SQLite (使用 `better-sqlite3`)
- **开发语言**: TypeScript (全栈类型安全)
- **测试框架**: Vitest (强制执行 TDD 流程)

## 架构设计要点

### 1. 文档即应用 (Docs as App)
利用 VitePress 不仅作为文档工具，更作为单页应用（SPA）的宿主。通过自定义主题和组件挂载业务逻辑，实现“文档站点”与“业务系统”的无缝融合。

### 2. 前后端共享常量
通过 `src/shared` 目录共享类型定义和业务常量（如网格尺寸 `30x40`、总点数 `1200`），确保前后端逻辑的一致性，避免映射错误。

### 3. 数据模型与预种子策略
- **数据规模**: 1200 点位 × 3 场景 × 5 次尝试 = 18,000 条初始记录。
- **初始化策略**: 系统启动时预种子化所有结果记录，以简化状态查询逻辑（无需动态创建记录，只需更新状态）。
- **表结构**:
  - `points`: 存储点位基础信息及备注。
  - `results`: 存储具体的验收尝试记录（点位 ID, 场景, 尝试次数, 是否选中）。

### 4. 测试驱动开发 (TDD)
计划严格遵循 TDD 流程：先写失败测试 -> 运行验证失败 -> 写最少实现代码 -> 运行验证通过。涵盖网格映射、进度计算、数据库初始化、API 行为及前端状态管理。

## 关键业务规则

- **网格映射**: 一维点位 ID (1-1200) 与二维坐标 (行/列) 的双向转换算法。
- **状态着色**: 根据完成度动态改变 UI 颜色：
  - `not-started`: 未开始
  - `in-progress`: 进行中
  - `in-progress-advanced`: 高级进行中 (覆盖 3 个场景)
  - `completed`: 已完成 (15 次尝试全部选中)
- **乐观更新**: 前端先更新 UI 状态，再异步发送请求，失败则回滚，以保证界面响应速度。

## 实施任务概览

1. **工程骨架**: 初始化项目配置、共享常量及网格映射工具。
2. **进度计算**: 实现点位状态判断逻辑及聚合函数。
3. **数据持久化**: 实现 SQLite 建表、种子数据初始化及仓储层。
4. **API 服务**: 构建原生 HTTP 服务，实现参数校验与 CRUD 接口。
5. **前端状态**: 实现客户端 API 封装、状态管理及自动保存协调。
6. **UI 组件**: 开发 VitePress 页面及核心交互组件（点阵、录入区、总览表）。
7. **部署交付**: 配置生产构建脚本，完成冒烟测试与部署文档。

## 矛盾与权衡

- **VitePress 定位**: 将静态文档工具用作动态应用容器，虽偏离常规用法，但适合内网工具场景，需注意 SEO 无关性及构建产物管理。
- **预种子数据**: 初始化 1.8 万条记录被视为数据冗余，但换取了查询逻辑的极大简化，适合单机中等规模数据场景。
- **版本控制**: 当前环境非 Git 仓库，采用逻辑里程碑而非实际提交，适合快速原型开发。

## 开放问题

- **数据备份**: 单机 SQLite 环境下的自动备份策略缺失。
- **并发处理**: 多人内网访问时的 SQLite 写入锁竞争问题。
- **扩展性**: 点位数量激增（如至 10,000）时，预种子策略和内存渲染的性能瓶颈。
