# BaseTrack Reference: Ultralytics Trackers Module

This source document serves as the authoritative API reference for the `ultralytics.trackers.basetrack` module. It defines the foundational architecture for Multi-Object Tracking (MOT) within the Ultralytics YOLO ecosystem. The module exposes two critical components: the `TrackState` enumeration and the `BaseTrack` abstract base class.

## Core Components

### TrackState Enumeration
The `TrackState` enum defines the lifecycle states of a tracked object. It is essential for understanding how the tracker manages objects through detection, occlusion, and removal. The states typically include:
- **New**: The object has just been detected and initialized.
- **Tracked**: The object is successfully being followed across frames.
- **Lost**: The object was not detected in the current frame but is retained in memory for potential re-identification (handling occlusion).
- **Removed**: The object has been lost for too long or exited the scene and is purged from the tracker's memory.

### BaseTrack Abstract Class
`BaseTrack` serves as the parent class for all specific tracking algorithms (e.g., Bot SORT, ByteTrack). It standardizes the interface and data schema for tracking objects, ensuring consistency across different implementations. Key responsibilities include:
- **Identity Management**: Assigning and maintaining unique track IDs.
- **State Management**: Updating the `TrackState` based on detection inputs.
- **History Preservation**: Storing historical bounding boxes, scores, and feature embeddings (for ReID) required for motion compensation and re-identification.

## Relevance to Existing Projects
This source clarifies the internal mechanics of the [[bot-sort-tracker]] previously documented. It explains *how* Bot SORT maintains identity during occlusion (via the `Lost` state) and where data like Kalman filter states and ReID embeddings are stored (within `BaseTrack` attributes). This is particularly relevant for the [[ai-gym-module]], where understanding the transition to the `Lost` state helps debug missed repetition counts when users are briefly occluded by equipment or other people.
