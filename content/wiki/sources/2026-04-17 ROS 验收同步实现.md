# 关键概念 (Key Concepts)

*   **ROS-Frontend Synchronization** (ROS-前端同步)
    *   **定义**: 通过 ROS 话题将机器人端的实时规划状态（点阵布局、区域进度）单向推送到 Web 前端，实现界面与物理世界的动态一致。
    *   **重要性**: 解决了硬编码点阵的问题，使前端能自适应不同的绑扎任务规划。
    *   **Wiki 状态**: 新概念，需创建页面。
*   **Dynamic Matrix Reconstruction** (动态矩阵重建)
    *   **定义**: 前端接收到全量规划 JSON 后，自动计算并更新点阵的行数、列数及每个点的全局索引，无需人工干预。
    *   **重要性**: 实现了 `grid-mapping` 的自动化，是“矩阵工作台”架构的核心演进。
    *   **Wiki 状态**: 扩展现有 `grid-mapping` 概念。
*   **Event-Driven Auto-Saving** (事件驱动自动保存)
    *   **定义**: 监听特定的 ROS 状态变更事件（如 `ready_for_next_area`），自动触发当前组数据的持久化操作。
    *   **重要性**: 减少了人工确认步骤，确保验收数据的实时性和完整性，符合 `optimistic-update` 的闭环。
    *   **Wiki 状态**: 新概念，需创建页面。
*   **Subagent-Driven Development** (子代理驱动开发)
    *   **定义**: 文档明确指定的开发方法论，要求 AI 代理将大计划拆解为带复选框的子任务，并逐个执行验证。
    *   **重要性**: 本项目的执行策略，强调测试先行（TDD）和自动化验证。
    *   **Wiki 状态**: 扩展现有 `test-driven-development` 概念。

# 主要论点与发现 (Main Arguments & Findings)

*   **核心主张**: 验收系统应从“静态配置 + 人工同步”演进为“动态规划同步 + 事件驱动自动保存”。
*   **证据支持**:
    *   计划详细定义了 ROS 端发布 `std_msgs/String` 携带 JSON 的具体 C++ 实现。
    *   前端通过 `parseAcceptancePlanJson` 将非结构化 JSON 映射为结构化 `AcceptancePlanState`。
    *   通过监听 `ready_for_next_area` 标志位实现无感保存，减少了人为遗漏风险。
*   **证据强度**: **高**。文档提供了完整的 TDD 流程（失败测试 -> 实现 -> 通过测试），包含具体的代码片段、SSH 验证命令和预期输出，具备极高的可执行性。

# 矛盾与张力 (Contradictions & Tensions)

*   **数据一致性风险**: ROS 端发布的是“规划”（Plan），而前端记录的是“验收结果”。如果机器人在运行过程中规划发生变更（重规划），前端如何处理已录入但属于旧规划的数据？文档未明确提及版本控制或规划变更时的冲突处理策略。
*   **网络依赖**: 方案强依赖 `wss://ros.cvalab.top` 或局域网连接。如果 ROS 话题发布延迟或中断，前端的“自动重建”和“自动保存”是否会进入错误状态或阻塞？文档提到了连接状态检查，但未详述断连后的降级策略。
*   **耦合度**: 前端直接依赖 ROS 话题名称（如 `/acceptance/plan`）和 JSON 结构。如果 ROS 侧数据结构调整，前端将直接失效，缺乏中间层的契约测试或版本协商机制。
