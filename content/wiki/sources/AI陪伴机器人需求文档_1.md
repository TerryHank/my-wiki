# AI 陪伴机器人需求文档_1

本文档定义了两人团队开发**静态毛绒 AI 玩偶**的最小可行性产品（MVP）规格。核心目标是快速构建一个具备触摸反馈、LCD 表情显示和语音对话能力的原型，严格限制功能范围以确保交付速度。

## 核心定义
项目被定义为“不会动的毛绒 AI 玩偶”，强调在静态外观下实现动态交互。
- **形态**：静态毛绒玩具，无机械运动部件。
- **交互**：单点触摸触发反馈，LCD 屏幕显示眼睛表情，语音双向对话。
- **架构**：采用 [[cloud-voice-architecture]]，设备端作为瘦客户端，所有智能处理（ASR/LLM/TTS）均在云端完成。

## 最小功能列表 (Top 5)
1. **触摸反馈**：检测到触摸后切换表情并播放预设回应。
2. **LCD 眼睛表情**：实现待机、眨眼、聆听、开心四种基础状态，由 [[expression-state-machine]] 控制。
3. **语音输入**：设备录音并上传至云端。
4. **AI 智能回复**：云端根据角色人设生成文本回复。
5. **语音播报**：将云端返回的语音在设备端播放。

## 硬件与软件边界
### 硬件配置
- **必需**：主控板、麦克风、扬声器、LCD 小屏（眼睛）、触摸传感器（单点）、Wi-Fi 模块、电池、蓝牙（预留）。
- **排除**：摄像头、多触摸区域、机械运动部件。

### 软件策略
- **记忆系统**：采用 [[weak-memory-strategy]]，仅存储用户昵称，不维护复杂上下文或长期记忆。
- **配网方案**：MVP 阶段开发版可写死 Wi-Fi 配置，暂不开发 App 配网流程。
- **用户系统**：暂不需要独立用户系统。

## 最小演示闭环
项目成功的唯一标准是跑通以下两个闭环：
1. **触摸闭环**：用户摸头 → 眼睛变开心 → 玩偶播放语音反馈。
2. **对话闭环**：用户说话 → 设备录音上传 → AI 生成回复 → 设备播报并切换表情。

## 团队分工
- **全栈设备工程师**：负责硬件选型、嵌入式开发、传感器集成及设备端逻辑（录音、播放、触摸检测、状态机）。
- **AI / 后端工程师**：负责云端 API 集成（ASR+LLM+TTS）、角色 Prompt 定义及最小后端接口开发。
