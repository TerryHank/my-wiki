# 矩阵验收工作台设计规格

**日期：** 2026-04-04

**目标：** 将当前验收录入系统重构为一套面向“异常登记”的矩阵工作台。前端继续采用 VitePress 承载，后端继续采用 Node.js + SQLite，系统运行在内网服务器 `10.0.0.99`，单机访问，无登录，只维护一份固定验收记录。

## 1. 背景与本次重构结论

上一版实现虽然已经具备基础保存能力，但交互模型仍然偏“录表”，不符合当前实际使用方式。用户已经明确新的工作规则：

- 页面必须保留右侧棋盘格矩阵总览
- 矩阵不是装饰图，而是主导航区
- 点位按蛇形遍历分组，每 `4` 个点为一组
- 左侧一次只处理当前组，且必须是严格 `2 x 2` 四宫格
- 每个点在当前场景下只有两种有效状态：`成功`、`失败`
- 默认状态视为 `成功`
- 鼠标单击点位可在 `成功 / 失败` 之间切换
- 失败时需要填写备注
- 备注按当前场景保存
- 点位恢复 `成功` 后，备注不删除，作为历史记录保留
- 矩阵平时只显示点；悬停时显示点号、二维坐标、当前场景备注

因此，本次重构不再延续“多次数测试方块”的旧模型，而是明确转为：

- 左侧四宫格异常工作台
- 右侧密集矩阵总览
- 默认成功、只记录异常

## 2. 范围与约束

### 2.1 已确认范围

- 使用 VitePress 作为前端页面框架与构建方式
- 使用 Node.js 提供轻量 API
- 使用 SQLite 作为持久化存储
- 系统部署在内网服务器 `10.0.0.99`
- 单机内网访问
- 无登录
- 只维护一份固定验收记录
- 支持动态新增场景，并在场景之间切换
- 支持用户输入总点数，矩阵根据点数自动生成
- 导出 Excel 时只导出当前场景

### 2.2 非目标

- 不支持多项目、多批次切换
- 不做账号体系、权限体系、操作审计
- 不做复杂流程审批
- 不保留旧版“每场景 5 次测试机会”的交互
- 不保留底部分页大表作为主录入界面
- 首版不处理多人同时编辑的冲突协调

## 3. 点位编号、坐标与蛇形规则

本系统不再写死 `40 x 30` 网格，而是由用户输入的总点数自动生成矩阵。

### 3.1 自动生成规则

- `pointCount` 由用户输入
- `columnCount = ceil(sqrt(pointCount))`
- `rowCount = ceil(pointCount / columnCount)`
- 目标是让矩阵尽量接近正方形，优先保持整体规整感

### 3.2 坐标定义

- 使用零基坐标
- 左上角坐标为 `00,00`
- `x` 轴从左到右递增
- `y` 轴从上到下递增
- 矩阵外圈显示坐标轴
- 坐标显示默认至少为两位数，例如 `00`、`01`、`09`
- 若未来坐标超过两位数，则自动扩展显示宽度

### 3.3 蛇形编号规则

点位编号按蛇形遍历顺序生成：

- 第 `0` 行（`y = 00`）从左到右
- 第 `1` 行（`y = 01`）从右到左
- 第 `2` 行（`y = 02`）从左到右
- 依此交替

即：

- 偶数行按 `x = 0 → columnCount - 1`
- 奇数行按 `x = columnCount - 1 → 0`

点号 `1` 永远位于左上角，即坐标 `00,00`。

### 3.4 分组规则

- 以蛇形遍历顺序为准，每 `4` 个点为一组
- 第 `1-4` 个点为第 `1` 组
- 第 `5-8` 个点为第 `2` 组
- 第 `9-12` 个点为第 `3` 组
- 依此递增

组号不按二维视觉位置切分，而按蛇形编号顺序切分。这一点必须保持一致，否则左侧四宫格和右侧矩阵会对不上。

### 3.5 末尾空位规则

当 `pointCount` 无法填满最后一行时：

- 最后一行允许存在空坐标位
- 空位只作为视觉占位，不可点击
- 空位不参与分组、状态统计和导出

## 4. 页面结构

页面采用固定的 `A` 方案：

- 左侧：当前组四宫格工作台
- 右侧：密集矩阵总览

顶部保留最少控制项：

- 当前场景切换
- 新增场景
- 总点数输入
- 当前场景 Excel 导出
- 自动保存状态提示

本版不再保留底部分页表格总览，避免界面再次回到“像在录 Excel”。

## 5. 左侧四宫格工作台

### 5.1 结构

左侧为固定宽度工作区，永远展示当前组的 `4` 个点，排布为严格 `2 x 2` 四宫格。

每个卡片包含：

- 点号
- 零基坐标
- 当前场景状态
- 备注编辑区或历史备注区

### 5.2 当前组行为

- 当前组由“当前点所在组”决定
- 点击右侧任一点，会直接切换到该点所在组
- 当前点在左侧四宫格中需要额外高亮
- `上一组 / 下一组` 按组切换，不按单点切换

### 5.3 状态交互

每个点的默认状态视为 `成功`。

交互规则：

- 单击一次：若当前为 `成功`，切换为 `失败`
- 再单击一次：若当前为 `失败`，切换回 `成功`
- 每次切换后立即自动保存

本版不引入 `未测试` 或 `待判定` 状态。系统只记录当前是否存在故障。

### 5.4 备注交互

- 备注与“当前场景 + 当前点”绑定
- 点位切为 `失败` 后，应立即暴露备注输入区
- 备注输入采用自动保存
- 点位恢复 `成功` 后，备注不清空
- 若当前点已恢复成功但存在历史备注，左侧卡片中应显示“历史备注”而不是直接丢失

### 5.5 左侧卡片的视觉表达

- `成功`：中性浅底或浅绿色底
- `失败`：浅红底
- 当前点：额外描边或阴影高亮
- 卡片不做大面积装饰，重点是清楚、稳定、好扫读

## 6. 右侧密集矩阵总览

### 6.1 视觉目标

右侧矩阵必须符合以下要求：

- 保持秩序感和密度感
- 平时只显示点，不常驻显示编号和坐标
- 矩阵是总览，不是左侧四宫格的放大复制
- 鼠标悬停时才显示详细信息

### 6.2 显示规则

矩阵中的每个有效点位只显示为一个点：

- `成功`：普通点
- `失败`：红点
- 当前组：外圈或描边高亮
- 当前点：比当前组更强的高亮

建议同时表达：

- 状态层：成功 / 失败
- 组层：当前组
- 焦点层：当前点

这三个层次不能互相覆盖掉。

### 6.3 悬停提示

鼠标悬停任一点时，弹出轻量提示，内容包括：

- 点号
- 二维坐标（零基）
- 当前场景名称
- 当前状态
- 当前场景备注

若该点当前场景下没有备注，则明确显示“无备注”。

### 6.4 点击行为

- 点击矩阵中的点，会切换当前点
- 左侧四宫格同步跳到该点所在组
- 若点击当前点，可直接触发状态切换，或仅切焦点，具体实现以降低误触为准

默认推荐：

- 单击矩阵点：切换焦点
- 在左侧卡片中切换成功 / 失败

如果后续验证发现效率不够，再增加“矩阵点直接切换状态”的快捷交互。

## 7. 场景模型

### 7.1 场景定义

- 系统支持动态新增场景
- 页面一次只工作在一个场景下
- 所有矩阵颜色、左侧状态、备注和导出都以“当前场景”为准

### 7.2 场景切换

- 切换场景后，右侧矩阵按该场景的故障记录重新着色
- 左侧四宫格同步切换为该场景下的状态和备注
- 当前点位置保持不变

## 8. 数据模型设计

本系统采用“默认成功、稀疏存储异常”的模型。

### 8.1 settings 表

建议字段：

- `id INTEGER PRIMARY KEY`
- `point_count INTEGER NOT NULL`
- `updated_at TEXT NOT NULL`

说明：

- `point_count` 为当前用户输入的总点数
- `column_count` 和 `row_count` 不强制落库，可按规则实时计算

### 8.2 scenes 表

建议字段：

- `id INTEGER PRIMARY KEY`
- `name TEXT NOT NULL`
- `created_at TEXT NOT NULL`

### 8.3 scene_point_records 表

建议字段：

- `point_id INTEGER NOT NULL`
- `scene_id INTEGER NOT NULL`
- `is_failed INTEGER NOT NULL DEFAULT 0`
- `note TEXT NOT NULL DEFAULT ''`
- `updated_at TEXT NOT NULL`

联合唯一键：

- `(point_id, scene_id)`

说明：

- 没有记录时，表示该点在该场景下默认为 `成功`
- 当点位被切成 `失败`，或填写了备注时，创建或更新记录
- 当点位恢复 `成功` 且备注仍保留时，记录继续保留
- 只有当 `is_failed = 0` 且 `note = ''` 时，才允许删除该记录以保持稀疏

### 8.4 不再保留的旧模型

以下旧字段和旧表语义不再使用：

- `attempt_index`
- 多次测试方块
- 场景下的 `5` 次机会模型

这部分数据不迁移进新模型。

## 9. API 设计

### 9.1 获取整份工作台数据

`GET /api/form`

返回内容：

- 当前设置（总点数、推导出的列数和行数）
- 全部场景列表
- 当前全部场景下的稀疏点位记录
- 各场景统计摘要

### 9.2 新增场景

`POST /api/scenes`

请求体示例：

```json
{
  "name": "阴雨天"
}
```

### 9.3 更新点位状态

`PUT /api/points/:id/state`

请求体示例：

```json
{
  "sceneId": 2,
  "isFailed": true
}
```

说明：

- 用于在 `成功 / 失败` 之间切换
- 切换后返回最新工作台数据，或返回该点及统计增量

### 9.4 更新点位备注

`PUT /api/points/:id/note`

请求体示例：

```json
{
  "sceneId": 2,
  "note": "亮斑异常，边缘拖尾"
}
```

### 9.5 导出当前场景 Excel

`GET /api/export.xlsx?sceneId=2`

导出列建议为：

- 点号
- 组号
- `X`
- `Y`
- 结果
- 备注
- 更新时间

说明：

- 只导出当前场景
- 结果只有 `成功`、`失败`
- 即使当前结果为 `成功`，若存在保留备注，也一并导出

## 10. 前端架构设计

建议结构：

- `components/AcceptanceApp.vue`：页面主组件
- `components/ProgressHeader.vue`：顶部控制区
- `components/PointInspector.vue`：左侧四宫格工作台
- `components/PointGrid.vue`：右侧密集矩阵
- `src/client/store.ts`：统一状态管理
- `src/shared/workbench.ts`：蛇形坐标、组号、矩阵尺寸计算
- `src/shared/types.ts`：共享类型

前端职责：

- 根据 `pointCount` 自动计算矩阵尺寸
- 在客户端维护当前点、当前组、当前场景
- 将稀疏记录映射为矩阵颜色
- 悬停时按当前场景显示备注 tooltip
- 将组导航和矩阵跳转保持一致

## 11. 后端架构设计

Node.js 服务负责两类能力：

- 提供静态资源
- 提供 SQLite API

后端要求：

- 启动时自动初始化数据库
- 严格校验 `pointId`、`sceneId` 和 `pointCount`
- 所有写操作更新 `updated_at`
- 以稀疏方式读写 `scene_point_records`
- 导出时补齐所有隐式成功点

## 12. 自动保存与错误处理

### 12.1 自动保存策略

- 状态切换：点击即保存
- 备注编辑：输入后短延迟保存
- 保存过程中显示轻量状态，例如“保存中”

### 12.2 失败处理

需要覆盖以下情况：

- 接口超时
- 参数越界
- 数据库锁冲突
- 服务器不可达

交互策略：

- 状态切换后先给界面反馈
- 请求失败时回滚状态或明确提示未保存
- 不允许界面显示为“已切换成功”但实际未落库
- 备注保存失败时，需要明确提示并保留输入内容

## 13. 样式与视觉方向

整体视觉采用“秩序型数据工作台”，不走花哨方向。

### 13.1 视觉原则

- 浅色背景，保证长时间录入不疲劳
- 左侧四宫格清晰、稳定、信息密度适中
- 右侧矩阵密集但不拥挤
- 强调秩序感，而非卡通感或装饰感
- 红色只用于失败点和异常备注

### 13.2 矩阵要求

- 点尽量小而清楚
- 坐标轴需清晰但不喧宾夺主
- tooltip 简洁，不遮挡过多内容
- 当前组和当前点的高亮表达要克制但可辨认

## 14. 性能要求

虽然点数可动态变化，但主页面仍需要保持流畅：

- 首屏应以稀疏记录渲染，不为每个点挂载重逻辑
- 矩阵悬停提示避免产生重排抖动
- 状态切换尽量只更新必要点位和左侧卡片
- 在 `pointCount` 较大时仍需保持可操作

## 15. 部署设计

目标服务器：`terry@10.0.0.99`

部署形态：

- 使用 VitePress 构建前端静态资源
- 使用 Node.js 运行 API 服务
- 使用 SQLite 保存数据文件

运行方式：

- Node 服务监听内网端口
- 静态资源与 API 由同一服务提供
- 继续使用国内源安装依赖

## 16. 测试与验收

### 16.1 前端验收点

- 输入总点数后，矩阵尺寸自动变化
- 矩阵按蛇形规则生成点号
- 左上角坐标为 `00,00`
- 当前组始终为严格 `2 x 2` 四宫格
- 点击某点后，左侧跳到正确组
- 失败点显示为红点
- 悬停红点时显示点号、坐标和当前场景备注
- 恢复成功后，历史备注仍可通过悬停看到

### 16.2 后端验收点

- 首次启动可自动初始化数据库
- 非法点号、非法场景、非法点数会被拒绝
- 状态切换后数据库记录正确
- 恢复成功但保留备注时，数据库记录不丢失
- 重启服务后数据不丢失

### 16.3 集成验收点

- 从场景切换到异常录入的主流程无阻塞
- 上一组 / 下一组切换正确
- 导出的 Excel 与当前场景一致
- 自动保存失败时有明确提示

## 17. 风险与取舍

### 17.1 默认成功的取舍

本版将“无记录”直接视为 `成功`。这极大简化了录入，但也意味着系统本质上更像“异常登记台”，而不是“逐点确认台”。如果未来需要区分“尚未确认”和“确认成功”，必须再引入第三种状态。

### 17.2 保留历史备注的取舍

恢复成功后保留备注，可以保住故障历史，但也意味着备注不再只代表“当前仍失败”。因此矩阵 tooltip 和导出都必须明确显示当前状态，避免误读。

### 17.3 动态矩阵的取舍

矩阵由点数自动生成，比固定 `40 x 30` 更灵活，但也要求前后端共享同一套尺寸计算和蛇形映射规则，否则定位会错位。

## 18. 最终结论

本系统采用 `A` 方案重构：

- 左侧固定四宫格工作台
- 右侧密集矩阵总览
- 矩阵由总点数自动生成
- 点号按蛇形规则编号
- 默认成功，只登记失败与备注
- 当前场景独立保存状态和备注

这套方案兼顾：

- 大规模点位的可导航性
- 故障录入效率
- 矩阵总览的秩序感
- 服务器持久化保存

后续实现必须严格以本规格为准，不再回退到旧版“多次数勾选表”的思路。
